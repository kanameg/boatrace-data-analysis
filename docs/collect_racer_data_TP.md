## 1. テスト概要

本テスト計画書は「ボートレーサーデータ収集プログラム 要求仕様書（SRS v1.0）」の実装品質を確保するための総合的なテスト戦略を定める。

**テストの主要目標：**
- 全ての機能要件（F-01～F-33）が正しく実装されることを検証
- 全ての非機能要件（NF-01～NF-11）が満たされることを検証
- エラーハンドリング・例外処理が想定通りに動作することを検証
- 出力データの整合性・正確性を検証
- 受入基準（AC-01～AC-16）をすべて満たすことを確認

---

## 2. テスト目的・スコープ

### 2.1 テスト目的

| 目的ID | 内容 |
|--------|------|
| TP-01 | 機能要件の実装検証：全F要件が設計通り動作することを確認 |
| TP-02 | 非機能要件の検証：言語・ライブラリ・エラーハンドリング・パフォーマンスを確認 |
| TP-03 | データ整合性：公式データから抽出・整形したCSVが正確であることを確認 |
| TP-04 | 互換性確認：出力CSVがExcel・pandas・他ツールで正常に読込み可能なことを確認 |
| TP-05 | 運用適性：スケジューラ実行・ログ出力・一時ファイル削除が正常に動作することを確認 |

### 2.2 テスト対象

**含める**
- Bashシェルスクリプト（`collect_racer_data.sh`）
- Pythonスクリプト（`src/process_racer_data.py`）
- CLI仕様（引数処理・終了コード・エラーメッセージ）
- ファイルシステム操作（ディレクトリ作成・ファイル削除）
- ログ出力仕様

**含めない**
- 公式サイトの仕様・データ品質
- Linux OSの基本機能
- 外部ツール（lhasa、pandas、numpy）の動作そのもの
- ネットワークインフラストラクチャ

### 2.3 テスト対象外の例外処理

以下は検証対象外であり、実装の仕様に委ねる：
- メモリ使用量の最適化
- CPU使用率
- 並列処理
- ユーザー権限管理

## 3. テスト戦略

### 3.1 テストアプローチ

**階層的テスト戦略**

```
単体テスト（ユニットテスト）
  ↓ 関数単位の正確性を検証
統合テスト
  ↓ モジュール間・Bash-Python間の連携を検証
システムテスト
  ↓ 全体フロー・実データでの動作を検証
受入テスト
  ↓ ユーザー観点での受入基準を検証
本番導入
```

### 3.2 テスト手法

| テスト手法 | 対象 | 実施方法 |
|----------|------|--------|
| **同値分割** | 入力値（年・期）、エラーコード | 有効/無効な入力値を分類 |
| **境界値分析** | 年の範囲（1990、実行年）、期値（1、2） | 最小/最大値の前後でテスト |
| **ステートメントカバレッジ** | Python関数 | `pytest --cov` で行カバレッジ確認（目標≥80%） |
| **エラーグエッシング** | 例外処理 | HTTP 404、タイムアウト、ファイルIOエラー等を意図的に発生させてテスト |
| **互換性テスト** | CSV出力 | Excel・pandas・LibreOfficeで読込み |
| **パフォーマンステスト** | 実行時間 | 実データで計測（目標≤10分） |

### 5.2 テスト環境セットアップ

**Dockerfile（開発環境と同一構成のテスト環境）**

```dockerfile
FROM python:3.12-slim

# Quality-of-life defaults for Python in containers.
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

RUN apt-get update && apt-get install -y \
    git \
    curl \
    lhasa \
    wget \
    bash-completion \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /workspaces
COPY requirements.txt /workspaces
RUN python -m pip install --upgrade pip && python -m pip install -r requirements.txt

RUN echo 'autoload bashcompinit && bashcompinit' >> ~/.bashrc && echo 'source /usr/share/bash-completion/completions/git' >> ~/.bashrc

ENV PATH="/root/.local/bin:$PATH"
```

**requirements.txt**

```

```

### 11.2 テストデータ例

**正常データ：2024年後期の選手情報（抜粋）**

```
racer_id,year,period,name,birthdate,grade,...
1,2024,2,塚本俊一,1969-11-11,A1,...
2,2024,2,栗田貴徳,1982-03-15,A2,...
```

## 14. リスク・対策

### 14.1 リスク分析

| リスクID | リスク内容 | 影響度 | 発生確率 | 対策 |
|----------|----------|--------|---------|------|
| R-01 | 公式サイトのデータレイアウト変更 | 高 | 低 | 定期的にレイアウト確認、SRS前提条件（P-01～P-03） |
| R-02 | ネットワーク環境不安定 | 中 | 中 | リトライロジック（F-12）、テスト環境でのネットワーク模擬 |
| R-03 | Python/pandasのバージョン非互換 | 中 | 低 | requirements.txt で固定、Docker環境での検証 |
| R-04 | lhasa がないテスト環境 | 中 | 低 | Docker で lhasa をインストール |
| R-05 | 実公式データの入手困難 | 低 | 低 | モックデータで代用、サンプル期のキャッシュ保持 |
| R-06 | テスト実行時間が10分を超える | 中 | 中 | AC-10 での計測、ボトルネック分析・最適化 |

### 14.2 対策

- **前提条件の監視**：SRS セクション9 の前提条件（P-01～P-03）に変更がないか定期確認
- **テスト環境の再現性**：Docker コンテナ化、requirements.txt 固定
- **リトライ・例外処理**：F-12 リトライロジック、エラーハンドリング仕様に従う
- **モックデータの準備**：ネットワーク不可の環境でも実行可能なテスト
